# Dev Log — March 11, 2026 (Research Studio backend + Sign in)

## Summary

Architected Research Studio as a two-repo setup (frontend on Vercel, backend on Heroku) following the Lingua Formula pattern. Created backend repo (`davidlong-tech-backend`), Flask app with auth, Postgres schema (auth + Research Studio tables), deployed to Heroku, ran migrations. Added Sign in link, sign-in/sign-out pages, studio placeholder, auth context, and Vercel API proxy to the frontend. Pushed to GitHub.

---

## Work Completed

### 1. Architecture and flight plan

- **Doc:** `docs/FLIGHT_PLAN.md` — phased implementation roadmap (Phase 1: backend; Phase 2: frontend integration; Phase 3: core loop).
- **Architecture:** Frontend (davidlong.tech) → Vercel; Backend (davidlong-tech-backend) → Heroku; Postgres on Heroku; auth via email/password in DB, JWT (cookie + Bearer).
- **Reference:** Lingua Formula (`/main_projects/python_projects/linguaformula`) used as architecture model.

### 2. Backend repo (`davidlong-tech-backend`)

- **Location:** `/main_projects/research_studio/davidlong-tech-backend` (sibling to davidlong.tech).
- **Stack:** Flask, gunicorn, psycopg2, bcrypt, PyJWT.
- **Structure:** `app.py`, `Procfile`, `requirements.txt`, `run.sh`, `migrations/`, `scripts/`.
- **venv:** Uses `venv/` (not `.venv`); `run.sh` creates venv, installs deps, starts Flask.
- **Health:** `GET /api/health` returns `{"ok": true, "service": "davidlong-tech-backend"}`.

### 3. Database setup

- **Local:** `scripts/create_database.sql` — creates `davidlong_tech` DB, grants `dev_user`. User ran as postgres superuser. Schema permissions granted via `GRANT ALL ON SCHEMA public TO dev_user`.
- **Migrations:** `001_add_user_table.sql`, `002_add_password_reset_table.sql`, `003_add_research_studio_schema.sql`, `004_seed_research_studio.sql`.
- **Schema:** `tbl_user` (email, password_hash, display_name), `tbl_password_reset`, `tbl_research_branches`, `tbl_research_mysteries`, `tbl_research_entries` (with `user_id` FK), `tbl_research_prompts`, `tbl_ai_operations`. Seed: 6 branches, 3 mysteries.
- **Scripts:** `scripts/run_migrations.sh`, `scripts/grant_schema_permissions.sql`.

### 4. Auth endpoints

- **Implemented:** `POST /api/auth/register`, `POST /api/auth/login`, `POST /api/auth/logout`, `GET /api/auth/me`.
- **Behavior:** JWT in cookie (`davidlong_tech_token`) and response body; 7-day expiry; `Authorization: Bearer` supported; bcrypt password hashing.
- **Tested:** Local and on Heroku.

### 5. Heroku deployment

- **App:** `davidlong-tech-backend` (URL: `https://davidlong-tech-backend-ddf83c56b82b.herokuapp.com`).
- **Add-on:** Heroku Postgres (essential-0).
- **Config:** `JWT_SECRET`, `CORS_ORIGINS`, `DATABASE_URL` (auto from add-on).
- **Migrations:** Run via `heroku run "psql \$DATABASE_URL -f migrations/..."`. Migrations 003 and 004 run together (Eco plan: 1 dyno limit).
- **Test user:** `verify@example.com` / `password123` (created during verification).

### 6. Frontend Sign in

- **Nav:** `SiteNav` shows "Sign in" when logged out; "Studio" and "Sign out" when logged in.
- **Pages:** `/sign-in` (email/password form, redirects to `/studio` on success), `/sign-out` (logs out, redirects to `/`), `/studio` (placeholder, requires auth; links to Today, Entries, Branches, Mysteries).
- **Auth:** `lib/authClient.ts` (getStoredToken, setStoredToken, clearStoredToken, authFetch), `context/AuthContext.tsx` (AuthProvider, useAuth, login, logout, refetch).
- **Proxy:** `vercel.json` rewrites `/api/:path*` → Heroku backend.
- **Local dev:** `NEXT_PUBLIC_API_URL` points to Heroku for API calls.
